# Deliver and Audit

Read this before claiming completion on Level 2–3 work. Only claim completion after every applicable check passes. If a check fails, say what remains and why.

## Checklist

### Scope

- [ ] Every explicit user request is answered.
- [ ] Requested name, format, path, and boundaries were respected.
- [ ] No unrequested structure was added that changes the nature of the deliverable.

### Evidence

- [ ] Claims are supported by sources, files, or observable facts.
- [ ] Assumptions are labeled. Uncertain points are visible.
- [ ] Unverifiable claims are removed or marked.

### Process

- [ ] Any failure was disclosed (see `failure-and-fallback.md`).
- [ ] Any fallback was disclosed with a reliability label.
- [ ] Required confirmations were actually obtained.
- [ ] If a specialized skill or tool was bypassed, the reason was stated.

### Usability

- [ ] The deliverable is complete and standalone — usable without reconstructing the conversation.
- [ ] No placeholders, no "same as above", no skeleton sections.
- [ ] Examples are realistic.

### Privacy

- [ ] No private identity, account data, secrets, local machine paths, or confidential project facts.
- [ ] Methods and generic examples are shared instead of private memory.

## Completion Statement

State plainly what was done and how it was verified. Do not pad with hedging, and do not claim verification that did not happen.

```text
Done: X and Y updated, Z verified by <method>. Remaining risk: <one line, or "none identified">.
```
